# 15.2 AI 文件/图片生成 — 核心功能

> **前置条件：15.1 前置改造已完成并通过全部测试。**
>
> 15.1 已完成：tool_executor 返回类型重构、llm.py tool 执行段拆分（streaming/non-streaming）、SSE 事件冒泡管道、ToolContext 传递、Go/前端 SSE 适配。

---

## 一、整体目标

为 AI 聊天系统新增文件/图片生成能力。通过扩展现有 Tool Call 机制，让 AI 在对话中自主生成图片和文件，生成的文件存入 MinIO 并作为 ResourceNode 自动出现在 Canvas 上，与产生它的 ChatNode 建立父子关系。

核心能力：
- **create_image**：AI 调用 GPT Image 1 (Medium) API，**流式生图**（partial_images 渐进预览），最终图写入 MinIO
- **create_file**：AI 直接输出文本格式文件（SVG、Markdown、CSV、JSON 等），存入 MinIO

---

## 二、核心设计决策

### 2.1 图片生成技术路线

采用**原生多模态输出 API**，不搭建代码执行沙箱：

| 方案 | 选择 | 理由 |
|------|------|------|
| 图片生成 | GPT Image 1 (Medium)，$0.07/张，流式 + partial_images | LM Arena 质量评分 ~1265（并列 #1）；OpenAI API 生态成熟，集成简单；Medium 档位兼顾质量与成本；流式生图支持渐进预览 |
| 文本格式文件 | LLM 直接输出内容 | SVG、MD、CSV、JSON、TXT 等本质是文本，LLM 可直接生成（不含 HTML，详见下方） |
| 代码执行沙箱 | 不实现（V1） | Excel/PPT 需要沙箱执行 Python 代码，复杂度高，后续版本再考虑 |

**V1 支持生成的文件类型：**

| 类型 | 生成方式 | 文件格式 |
|------|---------|---------|
| 图片 | GPT Image 1 (Medium)，1024×1024，流式 | PNG |
| SVG 矢量图 | LLM 直接输出 SVG 代码 | .svg |
| Markdown 文档 | LLM 直接输出 Markdown | .md |
| CSV 表格 | LLM 直接输出 CSV | .csv |
| JSON 数据 | LLM 直接输出 JSON | .json |
| 纯文本 | LLM 直接输出文本 | .txt |

**Tool 始终包含在 LLM 请求中**（与 web_search / url_reader 一致），由 LLM 自行判断是否使用。

**System Prompt 约束**：在 system prompt 中明确告知 LLM：「每次用户消息的回复过程中（包括所有中间 tool call 轮次），最多只能调用一次文件生成工具（`create_image` 或 `create_file`）。如果用户要求生成多个文件或图片，每次只生成一个，后续再引导用户逐个生成。」这是第一层防护（提示级），第二层在 llm.py 中通过 `file_gen_used` 标志位跨轮强制执行（见 §3.1.2 文件生成工具限制）。

### 2.3 流式图片生成（Streaming Image Generation）

**核心思路：利用 GPT Image 1 的 `stream=True` + `partial_images` 参数，在生图过程中向前端推送渐进预览，只有最终完整图才写入 MinIO。**

> **API 已验证可用**：GPT Image 1 的 `stream=True` + `partial_images=2` 已实测确认可正常工作。注意：每张 partial image 额外消耗 **100 image output tokens**（2 张 partial ≈ 200 额外 tokens）。
>
> **必须使用 `AsyncOpenAI` 客户端**：项目已有的 `_get_openai_client()` 返回 `AsyncOpenAI` 实例。流式生图需要 `await client.images.generate(stream=True, ...)` + `async for event in stream:`，而非同步迭代。

```
图片生成 ~10-20s
    │
    ├─ partial_image_0 (模糊预览)  →  JPEG q80 压缩 → SSE: image_partial {b64}  →  前端展示 base64
    ├─ partial_image_1 (更清晰)    →  JPEG q80 压缩 → SSE: image_partial {b64}  →  前端替换 base64
    └─ final_image (完整)          →  写 MinIO（原始 PNG，不压缩） → 注册 Go → SSE: resource_created {file_id, ...}
                                                                              →  前端把 src 从 base64 切换为 URL
```

**为什么不把 partial 也写 MinIO：**
- partial 是一次性的低质量预览，没有保存价值
- 减少 MinIO 写入次数（只写 1 次而不是 3 次）
- 前端收到 `resource_created` 后立即丢弃 base64，内存释放

**Partial 压缩策略：**
- GPT Image 1 返回的 partial 原始格式为 PNG，1024×1024 下可达 **1-4MB**
- 传输前先转 JPEG quality=80 压缩，压缩后 **~200-400KB**（缩小约 5-10 倍）
- 反正 partial 只是预览用途，JPEG q80 视觉质量与原始 PNG 几乎无差别
- 最终 final image 仍保持原始 PNG 写入 MinIO，不做压缩

**带宽评估（压缩后）：**
- 每张 partial image base64 ≈ **200-400KB**（1024×1024 JPEG q80 编码后）
- 2 张 partial ≈ **400-800KB** 走 SSE，完全可接受
- 经过两跳 SSE（ai-service → Go → 前端），无压力
- 压缩额外延迟 **~50-100ms**（PIL decode + JPEG encode），对 10-20s 生图过程可忽略

### 2.4 文件保存链路设计

**关键决策：ai-service 写 MinIO，Go backend 写 DB + 创建 Node/Edge。**

```
ai-service (tool_executor)
    │
    ├─ create_image（流式）:
    │   ├─ 调用 GPT Image API (stream=True, partial_images=2)
    │   ├─ 收到 partial → _compress_preview(q80) → yield ImagePartialEvent(base64)
    │   ├─ 收到 final → 写入 MinIO（仅 final，不压缩）
    │   ├─ 调用 Go backend 内部接口注册文件
    │   └─ yield ResourceCreatedEvent(file_id, node_id, ...)
    │
    └─ create_file（非流式）:
        ├─ 拿到 LLM 输出的文本内容
        ├─ 写入 MinIO（直接用 minio_client.write_file）
        ├─ 调用 Go backend 内部接口注册文件
        └─ 返回 (result_text, [ResourceCreatedEvent])

Go backend (内部接口)
    │
    ├─ 检查 24h 生成频率限制（Redis）
    ├─ 检查用户存储配额
    ├─ 创建 File 记录（写 DB）
    ├─ 创建 ResourceNode（写 DB）
    ├─ 创建 NodeEdge：ChatNode → ResourceNode（写 DB）
    └─ 返回 file_id + node_id + edge_id + position
```

**为什么 ai-service 直接写 MinIO 而不是传文件给 Go：**
- ai-service 已有 `minio_client.write_file()` 能力
- 避免在 HTTP 请求中传输图片二进制（可能数 MB），减少延迟
- ai-service 写 MinIO + Go 写 DB 的职责分离与现有 `file_convert_consumer` 模式一致

**为什么不复用现有 upload 接口：**
- upload 接口会触发 RabbitMQ → ai-service 文件转换，形成不必要的回流
- AI 生成的文件本身就是最终产物，不需要转换
- AI 生成的图片不做质量压缩，保持最佳体验（用户上传的图片才做压缩）

### 2.5 ResourceNode 反向父子关系

**现有模型**：Edge 的方向是 `Source → Target`，表示 Source 是 Target 的**父节点**（Source 为 Target 提供上下文）。

现有用法：`ResourceNode (source) → ChatNode (target)`，即文件为对话提供上下文。

**新增用法**：`ChatNode (source) → ResourceNode (target)`，即对话生成了这个文件。

当前 Node 模型和 NodeEdge 模型**无需任何 schema 变更**，因为 edge 本身已经是有向的，反向建边即可。只需要确保上下文组装时正确处理方向：

- `resolveParentContext()` 查询的是 `GetParentNodesByTargetID(conversationID)`，即 "谁指向我"
- 新增的 AI 生成 edge 方向是 `ChatNode → ResourceNode`，不会被 `resolveParentContext` 查到（`GetParentNodesByTargetID(chatNodeID)` 查的是 `target_node_id = chatNodeID` 的边，而新 edge 的 target 是 ResourceNode，不是 ChatNode，因此不会命中）
- 因此**不会产生循环引用**，也**不影响现有上下文组装逻辑**

**V1：AI 生成的图片不加入后续聊天上下文。** AI 生成的图片（`create_image`）不会作为后续对话的视觉上下文传入 LLM。tool result 中已包含足够的文本信息（文件名、file_id、"已展示给用户"），LLM 可据此在后续对话中引用该图片，无需重新传入图片二进制。这也避免了将大尺寸图片编码后塞入上下文导致的 token 消耗和延迟问题。后续版本（V2+）如需支持基于已有图片的多轮编辑，再引入图片上下文传递。

### 2.6 Go Backend 新增内部接口

**`POST /api/internal/ai/file`** — 供 ai-service await调用，注册 AI 生成的文件。

**请求体：**
```json
{
  "user_id": 12345,
  "canvas_id": 67890,
  "chat_node_id": "abc123nanoid",
  "message_id": "assistant_msg_nanoid",
  "minio_path": "users/12345/files/xyz_sunset.png",
  "filename": "sunset.png",
  "file_size": 204800,
  "content_type": "image/png"
}
```

**响应体**（遵循项目统一的 `dto.Success()` 包装格式）：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "file_id": "111222333444555",
    "node_id": "new_resource_node_id",
    "edge_id": "new_edge_id",
    "position": { "x": 600, "y": 400 },
    "image_url": "/api/file/111222333444555"
  }
}
```

> **注意**：`file_id` 为 int64 雪花 ID。为避免 JavaScript 大数精度丢失（`Number.MAX_SAFE_INTEGER` = 2^53），Go 端使用 `json:"file_id,string"` tag 将 int64 序列化为 string。前端用 `string` 类型接收。此规则适用于所有传输 `file_id` 的场景（Go 响应体、SSE 事件、前端类型定义）。
>
> **`canvas_id` 类型约定**：`canvas_id` 为 int64 雪花 ID，**仅在与前端交互时**使用 `json:",string"` tag 避免 JS 精度丢失；**后端内部通信**（Go → ai-service、ai-service → Go）一律使用 JSON integer，不加 `,string` tag。具体而言：`ToolContext.CanvasID`（Go → ai-service）和 `RegisterAIFileRequest.CanvasID`（ai-service → Go）均使用 `json:"canvas_id"`。~~15.1 中预定义的这两个 struct 已带 `json:"canvas_id,string"` tag~~ **已修正**：代码和 15.1 文档中的 `,string` tag 已移除。

**接口内部逻辑：**

```
1. 鉴权校验
   验证 X-Internal-Token
   校验 user_id 拥有 canvas_id 的权限（查询 canvas.user_id == user_id）
   校验 chat_node_id 属于该 canvas（查询 node.canvas_id == canvas_id AND node.id == chat_node_id）

2. 验证 24h 生成频率限制（Lua 脚本原子操作）
   执行 Redis Lua 脚本（先 GET 判断再 INCR，避免超限后仍递增计数器）：
   local current = redis.call("GET", KEYS[1])
   if current and tonumber(current) >= 10 then return -1 end
   local count = redis.call("INCR", KEYS[1])
   if count == 1 then redis.call("EXPIRE", KEYS[1], 86400) end
   return count
   如果返回 -1 → 返回 429（语义：-1 表示已达上限被拒绝）

3. 检查用户存储配额
   SUM(file_size) + new_file_size <= 200MB

4-8. **以下操作在同一个 DB 事务（tx）中执行，保证原子性：**

   4. 写入 File 记录
      INSERT INTO files (id, user_id, minio_path, filename, file_size, content_type)

   5. 查询 ChatNode 位置 + 已有子节点数量 → 计算新 ResourceNode 位置

   6. 创建 ResourceNode
      INSERT INTO nodes (id, canvas_id, node_type, pos_x, pos_y, file_id)

   7. 创建 NodeEdge
      INSERT INTO node_edges (id, canvas_id, source_node_id=chatNodeId, target_node_id=newResourceNodeId)

   8. tx.Commit()

9. 构造 image_url（仅图片类型）
   IF content_type starts with "image/" THEN
     image_url = "/api/file/{file_id}"
   ELSE
     image_url = ""
   **注意**：image_url 不在此接口中写入 messages 表，而是通过响应体返回给 ai-service → SSE 透传给 Go conversationService → 在 `complete` 事件写入 assistant message 时一并写入。详见 §2.16。

10. 返回 file_id, node_id, edge_id, position, image_url
```

**安全性**：
- 此接口通过内部网络访问（Docker 网络），不暴露给外部
- `X-Internal-Token` header 做简单鉴权
- Handler 或 Service 入口处校验 `user_id` 拥有 `canvas_id` 权限、`chat_node_id` 属于该 canvas，防止参数篡改

**孤儿文件清理策略**：ai-service 写入 MinIO 后若 Go 注册接口返回错误（429/507/500），ai-service 通过 `asyncio.create_task(_cleanup_minio_file_with_retry(path))` fire-and-forget 调度异步清理任务。清理函数内部使用 `asyncio.to_thread()` 包装同步 MinIO SDK 调用，并支持指数退避重试（最多 2 次重试，间隔 1s、2s）。清理过程不阻塞用户响应——用户端立即收到错误信息，清理在后台异步完成。若重试 3 次均失败则仅 log warning，V1 阶段可接受少量孤儿文件。

### 2.7 ResourceNode 自动定位

AI 生成的 ResourceNode 需要自动计算 Canvas 位置，避免与现有节点重叠。

**定位策略：相对于父 ChatNode 右下方偏移**

```
ChatNode (posX, posY)
    │
    └──→ ResourceNode_1 (posX + 400, posY + 100)
    └──→ ResourceNode_2 (posX + 400, posY + 250)
    └──→ ResourceNode_3 (posX + 400, posY + 400)
```

Go backend 在创建 ResourceNode 时：
1. 查询当前 ChatNode 已有的子 ResourceNode 数量 N（`WHERE source_node_id = chatNodeId`）
2. `posX = chatNode.PosX + 400`
3. `posY = chatNode.PosY + 100 + N * 150`

### 2.8 SSE 事件设计

> **注：SSE 事件的管道（ai-service → Go → 前端）已在 15.1 中搭建完成。15.2 只需让 tool executor 产生这些事件。**
>
> **⚠️ 15.2 SSE 格式变更**：15.1 中 `chat.py` 将 `image_partial` / `resource_created` 序列化为**扁平格式**（`{"type": "image_partial", "b64_image": "...", ...}`），15.1 代码与 15.1 文档一致，不存在偏差。**15.2 将格式改为嵌套格式**（`{"type": "image_partial", "data": {"b64_image": "...", ...}}`），以便 Go 端统一提取 `data` 字段透传，避免 `dto.SSEEvent` 包装后产生双层嵌套。此变更影响 ai-service `chat.py`、Go `aiClient.go` 透传逻辑，以及前端 `type.ts` 中 `type?: string` 兼容字段的处理。详见 §2.15.2 设计点 5 和部署注意事项。

#### 2.8.1 `image_partial` — 流式生图中间预览

**触发时机**：每收到一张 partial image，压缩为 JPEG q80 后立即推送。

```json
{
  "type": "image_partial",
  "data": {
    "b64_image": "/9j/4AAQSkZJRgABAQ...",
    "partial_index": 0,
    "chat_node_id": "abc123nanoid",
    "message_id": "assistant_msg_nanoid"
  }
}
```

**特点：**
- 只在 `create_image` 流式生成过程中产生
- partial_index 从 0 开始递增（通常 0 和 1）
- b64_image 为 JPEG q80 编码（~200-400KB），非原始 PNG
- `chat_node_id` 和 `message_id` 用于前端精确关联到对应消息，支持未来多图并发场景
- 前端收到后立即用 base64 渲染预览，后续 partial 替换前一张
- Go 端提取 `data` 字段后透传（仅解析外层 `type`/`data` 结构，不 re-parse 内部字段），详见 §2.15.2 设计点 5

#### 2.8.2 `resource_created` — 文件最终落库完成

**触发时机**：图片/文件写入 MinIO 并在 Go 注册完成后推送。

```json
{
  "type": "resource_created",
  "data": {
    "file_id": "111222333444555",
    "node_id": "new_resource_node_id",
    "edge_id": "new_edge_id",
    "filename": "sunset.png",
    "content_type": "image/png",
    "chat_node_id": "abc123nanoid",
    "message_id": "assistant_msg_nanoid",
    "position": { "x": 800, "y": 200 },
    "image_url": "/api/file/111222333444555"
  }
}
```

**特点：**
- `create_image` 和 `create_file` 都会产生此事件
- `image_url`：仅当 `content_type` 以 `"image/"` 开头时非空（覆盖 PNG 和 SVG），其余文件类型为空字符串
- Go `conversationService` 从此事件中提取 `image_url`，暂存后在 `complete` 时随 assistant message 一并写入 DB
- 前端收到后：在 Canvas 上创建正式 ResourceNode + Edge
- 对于图片：前端把 `<img src>` 从 base64 切换为 `/api/file/{file_id}` URL

#### 2.8.3 事件时序

**create_image（流式生图）：**

```
SSE: tool_call        {content: "Generating image: sunset landscape"}
SSE: image_partial    {b64_image: "...", partial_index: 0}       ← ~5s 模糊预览（JPEG q80）
SSE: image_partial    {b64_image: "...", partial_index: 1}       ← ~10s 更清晰（JPEG q80）
SSE: resource_created {file_id, node_id, edge_id, ...}          ← ~15s 最终图已入库
SSE: token            "I've created the image for you..."        ← LLM 第二轮回复
```

**create_file（非流式）：**

```
SSE: tool_call        {content: "Creating file: data.csv"}
SSE: resource_created {file_id, node_id, edge_id, ...}          ← 写 MinIO + 注册，~1-2s
SSE: token            "I've created the CSV file..."
```

### 2.9 Tool 执行结果返回给 LLM

Tool 执行成功后，返回简洁的结果文本给 LLM，让 AI 在回复中自然引用：

**create_image 成功：**
```
Image generated successfully.
- Filename: sunset.png
- File ID: 111222333444555
The image has been displayed to the user on the canvas.
```

**create_file 成功：**
```
File created successfully.
- Filename: report.md
- File ID: 111222333444555
The file has been displayed to the user on the canvas.
```

**失败（配额/限流）：**
```
Error: Daily file generation limit reached (10/10). Please try again tomorrow.
```
```
Error: Storage quota exceeded (200MB). Please delete some files and try again.
```

**失败（内容审核/政策违规）：**
```
Error: The image could not be generated because the prompt violates content policy. Please revise your description and try again.
```

**失败（超时）：**
```
Error: Image generation timed out (60s). Please try again.
```

**失败（文件过大）：**
```
Error: File content too large. Please request again.
```

**失败（不支持的文件类型）：**
```
Error: File type '.exe' is not supported. Allowed types: .csv, .json, .md, .svg, .txt.
```

LLM 收到成功结果后，会自然地在回复中提及 "I've generated the image/file for you"，无需额外格式化。LLM 收到错误信息后，会将错误原因告知用户。

### 2.10 生图费用追踪（Token 折算）

**问题**：`create_image` 的费用（$0.07/张 Medium）不体现在对话的 token 统计中，无法追踪成本。

**方案：将 `create_image` 费用折算为近似 token 数，一并存入 assistant message 的 `completion_tokens` 统计。**

GPT Image 1 (Medium) 实际消耗：
- **输入**：~275 input tokens（按 prompt 长度浮动）
- **输出**：~6,400 output tokens（1024×1024 medium 固定）
- **partial_images=2** 额外：200 output tokens（每张 partial 100 tokens）

**实现方式**：
- `create_image_stream` 成功后，从 `image_generation.completed` 事件的 `event.usage` 字段获取实际 token 消耗（`input_tokens` + `output_tokens`）
- 将此 token 数累加到当前轮次的 `TokenUsage` 中（与 LLM 文本生成的 token 合并），一起写入 assistant message
- ai-service 的 `stream_chat()` 在 `create_image_stream` 返回后，从 generator 获取的最终 usage 信息累加到总 `TokenUsage`

**token 累加逻辑**（在 llm.py streaming tool 分支中）：
```python
for block in streaming_tools:
    result_text = ""
    async for event in tool_executor.create_image_stream(block.input, tool_context):
        if isinstance(event, str):
            result_text = event
        elif isinstance(event, ImageGenUsage):
            # 累加图片生成的 token 到总量
            total_prompt_tokens += event.input_tokens
            total_completion_tokens += event.output_tokens
        else:
            yield event
    tool_results.append(...)
```

> **注意**：GPT Image API 的 token 定价（$0.01/1K output tokens for gpt-image-1）与 LLM 文本 token 定价不同，但统一折算为 token 数方便展示和统计。精确成本分析需后续在计费模块中区分 token 来源。
>
> **ImageGenUsage 仅在 `create_image` 流式分支中产生**（只使用 GPT Image 1 Medium 一种生图模型），不涉及三个 LLM provider 各自处理。llm.py 中三个 provider 函数（`_stream_chat_claude`、`_stream_chat_openai`、`_stream_chat_gemini`）各自有独立的 streaming tool 分支代码（结构一致但非共享函数），三处均需添加 `ImageGenUsage` 累加逻辑。

### 2.12 前端 Canvas 事件处理

收到 `resource_created` SSE 事件后，前端需要：

1. **在 Canvas 上创建 ResourceNode**：
   - 使用事件中的 `node_id`、`position`、`file_id` 创建节点
   - 与现有 ResourceNode 渲染逻辑复用

2. **创建 Edge**：
   - `source: chat_node_id`，`target: node_id`
   - 边的类型 `custom-edge`，与现有 edge 一致

3. **更新 Redux 状态**：
   - dispatch `addGeneratedResource`（已在 15.1 定义）
   - 同时更新 `nodes` 和 `edges`

**重要：`addGeneratedResource` 不写入 pendingDelta。** Go 内部接口（`POST /api/internal/ai/file`）已经将 ResourceNode 和 NodeEdge 写入数据库，前端只需更新本地 Redux 状态用于渲染，**不需要再通过 pendingDelta sync 回后端**。如果写入 pendingDelta，sync 时 `tx.Create()` 会因同 ID 已存在触发 `ErrDuplicatedKey` 错误。

注意：`image_partial` 事件**不操作 Canvas**，只在消息区域展示 base64 预览。Canvas 上的 ResourceNode 仅在 `resource_created` 时才正式创建。

#### 2.12.1 ResourceNode Handle 设计

**问题**：现有 ResourceNode 只有 `source` handle（右侧），用于 `ResourceNode → ChatNode` 方向的 edge（用户上传文件提供上下文）。AI 生成文件的 edge 方向是 `ChatNode → ResourceNode`，需要 ResourceNode 有 `target` handle 来锚定连线。

**约束**：
- 用户**不能**手动拖拽 edge 到 ResourceNode 的 target handle（ChatNode→ResourceNode 只能由后端创建）
- 当 incoming edge 被删除后，target handle 应视觉消失，回到"只有 source handle"的状态
- 后端 `node_type` 保持 `"resourceNode"` 不变，**无需新增节点类型**

**方案：ResourceNode 始终渲染 target handle（DOM 存在保证 edge anchor 不丢失），但默认隐藏 + 不可交互，仅在有 incoming edge 时可见。**

```tsx
// ResourceNode.tsx 改动
import { Handle, Position, useHandleConnections } from "@xyflow/react";

function ResourceNode({ id, data, selected }) {
  const hasIncomingEdge = useHandleConnections({ type: "target" }).length > 0;

  return (
    <div ...>
      {/* Header 根据 incoming edge 条件渲染 */}
      <span className="text-xs font-bold uppercase tracking-wider text-secondary">
        {hasIncomingEdge ? "Generated" : "Source"}
      </span>

      {/* ... 现有文件预览内容不变 ... */}

      {/* Target handle: 始终渲染（保证 edge anchor），但默认隐藏 + 不可交互 */}
      <Handle
        type="target"
        position={Position.Left}
        isConnectable={false}
        className={`custom-handle custom-handle-target ${hasIncomingEdge ? "" : "handle-hidden"}`}
        style={{ top: 20, left: -8 }}
      />

      {/* Source handle: 始终存在，正常交互 */}
      <Handle
        type="source"
        position={Position.Right}
        className="custom-handle custom-handle-source"
        style={{ top: 20, right: -8 }}
      />
    </div>
  );
}
```

```css
/* index.css 新增 */
.handle-hidden {
  opacity: 0;
  pointer-events: none;
  /* 禁用 pulse 动画、hover label 等所有视觉效果 */
}
```

**为什么 target handle 始终在 DOM 中（而非条件渲染）：**
- `addGeneratedResource` 同时添加 node + edge，首次渲染时若条件渲染 `{hasIncomingEdge && <Handle>}`，handle 可能尚未挂载 → edge 找不到 anchor → 连线位置异常
- 始终渲染 + CSS 隐藏避免时序问题，React Flow 始终能找到 target handle 的 DOM 元素

**行为总结：**

| 场景 | Target Handle | Source Handle | Header |
|------|--------------|---------------|--------|
| 用户上传的 ResourceNode | 隐藏 + 不可交互 | 正常（可拖拽创建 edge） | "Source" |
| AI 生成的 ResourceNode（有 incoming edge） | 可见 + 不可交互（`isConnectable={false}`） | 正常 | "Generated" |
| AI 生成的 ResourceNode（incoming edge 被用户删除） | 隐藏 + 不可交互 | 正常 | "Source" |

### 2.13 存储与限流策略

| 限制项 | 值 | 实现方式 |
|--------|-----|---------|
| 24h 文件生成上限 | 10 次 | Redis Lua 脚本原子操作（INCR + 条件 EXPIRE 86400），**固定窗口**：从用户首次生成时刻起算 24h |
| 用户总存储配额 | 200 MB | 复用现有 `maxStoragePerUser` 检查 |
| AI 生成图片质量 | 不压缩 | 原始质量保存，给用户最好的体验 |
| 单张图片大小 | 不限制（API 返回多大就存多大） | 200MB 配额本身是约束 |
| 单个文本文件大小 | 11MB 硬上限 | 防御性兜底，tool description 提示 AI 不超过 10MB |
| 每次用户轮次最多文件生成次数 | 1 | `create_image` 和 `create_file` 合计最多 1 次（跨轮累计，从用户发一条消息到 assistant 最终回复的完整轮次），tool description 提示 + llm.py `file_gen_used` 标志位跨轮强制执行 |
| 图片生成 API 超时 | 60s | 图片生成比搜索慢，超时放宽 |

**不压缩 AI 生成的图片**（区别于用户上传的图片会压缩到 1568px JPEG）：
- 保持最佳质量体验
- 200MB 配额 + 10 次/24h 的双重限制已经足够防滥用
- 不压缩意味着图片可能数 MB，反而加速了配额消耗，间接限制了使用量

### 2.14 错误处理与回滚策略

**核心原则：每个步骤失败后，清理已完成的操作，向 LLM 返回用户友好的错误信息。不做重试，直接返回错误。**

#### 2.14.1 `create_image` 错误处理流程

```
create_image_stream() 执行链路：

Step 1: 调用 GPT Image 1 API
    ├─ 成功 → 继续
    ├─ content_policy_violation (BadRequestError) → yield 错误文本，结束
    ├─ API 超时 (APITimeoutError, 60s) → yield 超时错误文本，结束
    └─ 其他 API 错误 → yield 通用错误文本，结束

Step 2: 写入 MinIO
    ├─ 成功 → 继续
    └─ 失败 → yield 通用错误文本，结束（无需回滚，文件未写入）

Step 3: 调用 Go 注册接口
    ├─ 成功 → yield ResourceCreatedEvent + tool result
    ├─ 429 (频率限制) → 清理 MinIO 文件 → yield 限流错误文本
    ├─ 507 (配额不足) → 清理 MinIO 文件 → yield 配额错误文本
    └─ 其他错误 → 清理 MinIO 文件 → yield 通用错误文本
```

#### 2.14.2 `create_file` 错误处理流程

```
create_file() 执行链路：

Step 0: 文件名清洗 + 白名单校验 + 大小检查
    ├─ 通过 → 继续
    ├─ 扩展名不在白名单 → 返回 ("Error: File type not supported", [])
    └─ 内容超过 11MB → 返回 ("Error: File content too large. Please request again.", [])

Step 1: 写入 MinIO
    ├─ 成功 → 继续
    └─ 失败 → 返回 ("Error: ...", [])

Step 2: 调用 Go 注册接口
    ├─ 成功 → 返回 (result_text, [ResourceCreatedEvent])
    ├─ 429/507 → 清理 MinIO 文件 → 返回 ("Error: ...", [])
    └─ 其他错误 → 清理 MinIO 文件 → 返回 ("Error: ...", [])
```

#### 2.14.3 回滚策略总结

| 失败步骤 | 已完成操作 | 回滚动作 |
|---------|-----------|---------|
| GPT Image API 失败 | 无 | 无需回滚 |
| MinIO 写入失败 | 无（或 API 已调用但已计费） | 无需回滚（API 费用不可退） |
| Go 注册接口失败 | MinIO 已写入 | ai-service 通过 `asyncio.create_task(_cleanup_minio_file_with_retry())` fire-and-forget 异步清理（退避重试 2 次），不阻塞用户响应 |
| asyncio.CancelledError | MinIO 可能已写入（`to_thread` 边界情况） | 只要 `minio_path` 非空即尝试清理（即使 `minio_written=False`），因为 `to_thread` 内部可能已完成写入但 Python 层 await 被取消 |

**不做重试**：所有步骤失败后直接返回错误文本给 LLM，LLM 会将错误原因告知用户。用户可自行重试。

### 2.15 客户端断开连接 Cancel 机制

**问题**：用户在流式生图过程中离开页面或手动取消，`create_image_stream` 仍会继续调用 GPT Image API 直到完成，浪费 API 费用和服务器资源。

**方案：后台协程轮询 `Request.is_disconnected()`，设置 `context.cancelled` 标志，`create_image_stream` 在安全检查点检测该标志提前终止。**

#### 2.15.1 整体架构

```
客户端断开 SSE 连接
    │
    ▼
Starlette ASGI 层检测到 http.disconnect
    │
    ▼
_monitor_disconnect 后台协程（每秒轮询 request.is_disconnected()）
    │
    ├─ 检测到断开 → context.cancelled = True
    │
    ▼
create_image_stream 在每个 partial 事件间检查 context.cancelled
    │
    ├─ True → yield 错误文本，提前 return（不再调 MinIO / Go）
    └─ False → 继续正常流程
```

#### 2.15.2 chat.py 实现

```python
import asyncio
from starlette.requests import Request

@router.post("/chat/completions")
async def stream_chat(request: Request, req: StreamChatRequest):
    # ... messages 构建 ...

    tool_context = None
    if req.tool_context:
        tool_context = ToolContext(
            user_id=req.tool_context.user_id,
            canvas_id=req.tool_context.canvas_id,
            chat_node_id=req.tool_context.chat_node_id,
            message_id=req.tool_context.message_id,
        )

    async def _monitor_disconnect():
        """后台协程：每秒轮询客户端连接状态，断开时设置 cancelled 标志。"""
        while tool_context and not tool_context.cancelled:
            if await request.is_disconnected():
                tool_context.cancelled = True
                logger.info("Client disconnected, setting cancelled flag")
                return
            await asyncio.sleep(1)

    async def event_generator():
        # 仅当 tool_context 存在时启动 disconnect 监控（15.1 阶段 tool_context 为 None，不启动）
        watcher = None
        if tool_context:
            watcher = asyncio.create_task(_monitor_disconnect())
        try:
            usage: TokenUsage | None = None
            async for item in llm.stream_chat(messages, req.model, tool_context):
                if isinstance(item, ImagePartialEvent):
                    # 15.2 变更：嵌套格式，与 §2.8.1 一致
                    yield {
                        "data": json.dumps({
                            "type": "image_partial",
                            "data": {
                                "b64_image": item.b64_image,
                                "partial_index": item.partial_index,
                                "chat_node_id": item.chat_node_id,
                                "message_id": item.message_id,
                            },
                        }),
                    }
                elif isinstance(item, ResourceCreatedEvent):
                    # 15.2 变更：嵌套格式，与 §2.8.2 一致
                    yield {
                        "data": json.dumps({
                            "type": "resource_created",
                            "data": {
                                "file_id": item.file_id,
                                "node_id": item.node_id,
                                "edge_id": item.edge_id,
                                "filename": item.filename,
                                "content_type": item.content_type,
                                "chat_node_id": item.chat_node_id,
                                "message_id": item.message_id,
                                "position": item.position,
                                "image_url": item.image_url,
                            },
                        }),
                    }
                # ... 其余 isinstance 分支（TokenUsage / ToolCallEvent / str）不变 ...
            # ... yield complete event ...
        except Exception as e:
            logger.exception("Stream chat error")
            yield {"data": json.dumps({"type": "error", "message": str(e)})}
        finally:
            if watcher:
                watcher.cancel()

    return EventSourceResponse(event_generator())
```

**关键设计点：**

1. **`_monitor_disconnect()` 作为独立的后台 `asyncio.Task`**：不阻塞主生成器流程，每秒轮询一次 `request.is_disconnected()`。1 秒的轮询间隔对 10-20 秒的生图过程来说足够及时。

2. **仅当 `tool_context` 存在时启动**：15.1 阶段 `tool_context` 为 `None`，不会创建后台任务，零开销。**注意**：15.2 上线后 `tool_context` 将始终非 `None`（Go 端每次请求都会传入），因此 `_monitor_disconnect` 将在**所有对话**中启动（包括不涉及文件生成的普通对话）。其开销极低（每秒一次 `is_disconnected()` 调用，仅检查 ASGI receive channel 是否有 disconnect 消息），且为普通对话也提供了断开检测能力（未来可扩展），属可接受的 trade-off。如需进一步优化，可改为仅在检测到文件生成 tool call 后再延迟启动，但 V1 不做此优化。

3. **`finally` 中 `watcher.cancel()`**：生成器正常结束或异常退出时都会取消后台轮询任务，避免泄漏。

4. **`request.is_disconnected()` 是 Starlette 原生 API**：通过 ASGI `receive` channel 检测 `http.disconnect` 消息，可靠且无依赖。

5. **`image_partial` / `resource_created` 嵌套格式（15.2 变更）**：ai-service 发送的 JSON 使用 §2.8 定义的嵌套格式 `{"type": "...", "data": {...}}`。Go 端 `aiClient.go` 中对这两类事件的透传逻辑需相应调整：从直接使用 `json.RawMessage(rawData)` 改为提取内层 `data` 字段后再存入 `AIStreamEvent.RawData`，确保经 `dto.SSEEvent{Type, Data}` 包装后序列化结果与 §2.8 完全一致，不会产生双层嵌套。其余事件（`token`、`tool_call`、`complete`、`error`）保持原有扁平格式不变（Go 端对这些事件使用 typed struct 完整解析后重新包装）。

#### 2.15.3 create_image_stream 中的检查点

`create_image_stream` 在两个安全位置检查 `context.cancelled`：

```python
async for event in stream:
    # 检查点：每次收到 partial 事件前检查（GPT Image API 两次 partial 间隔约 5 秒）
    if context.cancelled:
        logger.info("create_image cancelled by client disconnect")
        yield "Error: Image generation was cancelled."
        return

    if event.type == "image_generation.partial_image":
        # ... 压缩 + yield ImagePartialEvent ...
    elif event.type == "image_generation.completed":
        # ... 保存 final image ...

# 检查点：写入 MinIO 前检查（避免不必要的存储写入）
if context.cancelled:
    yield "Error: Image generation was cancelled."
    return

image_bytes = base64.b64decode(final_b64)
# ... 写入 MinIO + 调用 Go ...
```

**为什么只在 partial 间和 MinIO 写入前检查（不在 MinIO 写入后检查）：**
- MinIO 写入后 + Go 注册前取消 → 会产生 MinIO 孤儿文件。但此时生图已完成（API 已计费），不如让流程走完注册。即使前端已断开，文件注册完成后用户重新加载页面仍能看到文件。
- 因此 **MinIO 写入后不再检查 cancelled**，让注册流程完整执行。

#### 2.15.4 与 asyncio.CancelledError 的关系

`context.cancelled` 是**协作式取消**（cooperative cancellation），提供受控的提前退出。此外，sse-starlette 在客户端断开时也可能通过 task cancellation 触发 `asyncio.CancelledError`，但时机不可控（可能在 MinIO 写入中途）。

`create_image_stream` 的 `except Exception` 已覆盖 cleanup 逻辑。`asyncio.CancelledError` 在 Python 3.9+ 不继承 `Exception`（继承 `BaseException`），因此需要**额外**捕获：

```python
except asyncio.CancelledError:
    # Task 被取消（sse-starlette 主动取消），清理可能已写入的 MinIO 文件
    # 即使 minio_written=False 也尝试清理（to_thread 内部可能已完成写入）
    if minio_path:
        asyncio.create_task(_cleanup_minio_file_with_retry(minio_path))
    raise  # 重新抛出，让 asyncio 正常处理取消
```

### 2.16 消息图片持久化（image_url 字段）

**问题**：用户离开对话后重新载入，消息中生成的图片无法显示。前端 `message.metadata.generatedFiles` 仅存在于 Redux 内存状态（通过 SSE 事件收集），不持久化到数据库。

**方案：messages 表新增 `image_url` 字段，存储 AI 生成的图片类文件的访问 URL。**

> **`image_url` 用于所有图片类型文件**：包括 `create_image` 生成的 PNG 和 `create_file` 生成的 SVG（`image/svg+xml`），因为 SVG 可以被 `<img>` 标签同等展示。非图片类型的 `create_file` 产物（CSV、JSON、MD、TXT）不写入 `image_url`，用户通过 Canvas 上的 ResourceNode 查看/下载。每条 assistant message 最多生成一个文件（tool description + `llm.py` 强制限 `create_image` + `create_file` 合计 1 次）。

```sql
ALTER TABLE messages ADD COLUMN image_url TEXT DEFAULT NULL;
-- 存储格式：单个 URL 字符串，如 "/api/file/111222333444555"
-- 写入条件：content_type 以 "image/" 开头（覆盖 create_image 的 PNG 和 create_file 的 SVG）
-- 每条 assistant message 最多一个 AI 生成的图片类文件
```

**写入时机（方案 B：SSE 透传 + complete 时写入）**：

Go 内部接口 `RegisterAIGeneratedFile` **不直接 UPDATE messages 表**（因为 assistant message 行在流式传输期间尚未被 INSERT，UPDATE 会静默影响 0 行导致 image_url 丢失）。改为：

1. Go 内部接口在响应体中返回 `image_url` 字段（仅当 `content_type` 以 `"image/"` 开头时返回非空值，如 `"/api/file/111222333444555"`；其余返回空字符串）
2. ai-service 将 `image_url` 放入 `ResourceCreatedEvent`，通过 SSE 透传给 Go `conversationService`
3. Go `conversationService` 在处理 `resource_created` 事件时，从 `RawData` 中提取 `image_url`，暂存到局部变量 `capturedImageURL`
4. 在 `case "complete":` 写入 assistant message 时（`s.conversationRepo.CreateMessage`），将 `capturedImageURL` 一并写入 `ImageURL` 字段
5. 在 `writeErrAssistant` 闭包中同样写入 `capturedImageURL`（即使流式中断/出错，已成功生成的图片仍需持久化）

这样 `image_url` 随 assistant message 的创建一起写入，不存在时序问题。

**读取时机**：前端加载对话历史时，从 `message.image_url` 渲染图片卡片，无需依赖 SSE 事件重放。

**Go 端变更**：
- `RegisterAIGeneratedFile` 响应体新增 `image_url` 字段（由 Service 方法根据 contentType 计算）
- **⚠️ `SendMessage` 和 `RetryMessage` 两个函数都必须实现以下四步，缺一不可：**
  1. 函数开头声明 `var capturedImageURL string` 局部变量
  2. `case "resource_created":` 中从 `event.RawData` JSON 反序列化提取 `image_url` 字段，赋值给 `capturedImageURL`
  3. `case "complete":` 创建 assistant message 时写入 `ImageURL: ptrIfNonEmpty(capturedImageURL)`（`ptrIfNonEmpty` 仅当非空时返回 `*string`）
  4. `writeErrAssistant` 闭包中同样写入 `ImageURL: ptrIfNonEmpty(capturedImageURL)`（即使流式中断/出错，已成功生成的图片仍需持久化，不能丢失）

**前端变更**：
- `AssistantMessage.tsx` 渲染逻辑（三态优先级）：
  1. **流式生成中**：如果 `imagePreviews[messageId]` 存在 → 渲染 `<img src="data:image/jpeg;base64,...">` + loading 动画
  2. **流式完成后**：如果 `message.metadata.generatedFiles` 非空 → 渲染正式文件卡片（`src` 为 `/api/file/{fileId}`）
  3. **加载历史消息**：如果 `message.image_url` 非空 → 直接渲染 `<img src={message.image_url}>` 文件卡片（无需 SSE 事件重放）
- `image_url` 为单个 URL 字符串（如 `"/api/file/111222333444555"`），前端直接作为 `<img src>` 使用（PNG 和 SVG 均可被 `<img>` 标签渲染）
- 流式阶段仍通过 SSE 事件驱动（image_partial → resource_created）

**`create_image_stream` — async generator，通过 15.1 搭建的事件冒泡管道向前端推送事件：**

```python
async def create_image_stream(
    arguments: dict, context: ToolContext
) -> AsyncGenerator[ImagePartialEvent | ResourceCreatedEvent | ImageGenUsage | str, None]:
    """
    流式生图：yield ImagePartialEvent → yield ResourceCreatedEvent → yield str (tool result) → yield ImageGenUsage。
    最后 yield 的 str 作为 tool result 喂回 LLM，ImageGenUsage 用于 token 费用追踪。

    注意：ImageGenUsage 不属于 ToolSideEvent（不经 SSE 推送给前端），仅供 llm.py 内部
    累加到 TokenUsage 统计中。llm.py 的 streaming tool 分支通过 isinstance(event, ImageGenUsage)
    单独处理，不走 yield event 冒泡路径。
    """
    prompt = arguments["prompt"]
    filename = _sanitize_filename(arguments["filename"])

    minio_path = ""
    minio_written = False  # 标记 MinIO 是否已成功写入，仅在 True 时执行孤儿文件清理

    try:
        # 1. 流式调用 GPT Image 1 API（必须使用 AsyncOpenAI 客户端）
        # 注意：_get_openai_client 定义在 llm.py 中，而 llm.py 反向 import tool_executor，
        # 存在循环依赖。使用函数内 lazy import 打破循环：
        from services.llm import _get_openai_client
        openai_client = _get_openai_client("openai")
        stream = await openai_client.images.generate(
            model="gpt-image-1",
            prompt=prompt,
            n=1,
            quality="medium",
            size="1024x1024",
            stream=True,
            partial_images=2,
            timeout=60,  # §2.13: 图片生成 API 超时 60s（AsyncOpenAI 默认 600s，需显式覆盖）
        )

        final_b64 = ""
        image_usage = None  # 用于费用追踪
        async for event in stream:
            # Cancel 支持：在每次 partial 事件间检测连接是否已断开
            # context.cancelled 由上层 SSE handler 在客户端断开时设置
            if context.cancelled:
                logger.info(f"create_image cancelled by client disconnect")
                yield "Error: Image generation was cancelled."
                return

            if event.type == "image_generation.partial_image":
                # 原始 PNG 可达 1-4MB，压缩为 JPEG q80 后 ~200-400KB
                compressed_b64 = _compress_preview(event.b64_json, quality=80)
                yield ImagePartialEvent(
                    b64_image=compressed_b64,
                    partial_index=event.partial_image_index,
                    chat_node_id=context.chat_node_id,
                    message_id=context.message_id,
                )
            elif event.type == "image_generation.completed":
                final_b64 = event.b64_json  # final 保持原始 PNG，不压缩
                image_usage = event.usage    # {input_tokens, output_tokens}

        if not final_b64:
            yield "Error: Image generation failed — no final image received."
            return

        # Cancel 检查点 2：写入 MinIO 前检查（避免不必要的存储写入）
        if context.cancelled:
            logger.info("create_image cancelled before MinIO write")
            yield "Error: Image generation was cancelled."
            return

        # 2. 仅 final image 写入 MinIO（不压缩）
        # minio_client.write_file 是同步阻塞调用，需用 asyncio.to_thread 避免阻塞事件循环
        image_bytes = base64.b64decode(final_b64)
        safe_name = f"{nanoid()}_{filename}.png"
        minio_path = f"users/{context.user_id}/files/{safe_name}"
        await asyncio.to_thread(minio_client.write_file, minio_path, image_bytes, "image/png")
        minio_written = True  # MinIO 写入成功，后续失败需要清理

        # 3. 调用 Go backend 注册
        result = await _register_ai_file(
            user_id=context.user_id,
            canvas_id=context.canvas_id,
            chat_node_id=context.chat_node_id,
            message_id=context.message_id,
            minio_path=minio_path,
            filename=f"{filename}.png",
            file_size=len(image_bytes),
            content_type="image/png",
        )

        # 4. yield 最终事件
        yield ResourceCreatedEvent(
            file_id=result["file_id"],  # Go 端返回 string
            node_id=result["node_id"],
            edge_id=result["edge_id"],
            filename=f"{filename}.png",
            content_type="image/png",
            chat_node_id=context.chat_node_id,
            message_id=context.message_id,
            position=result["position"],
            image_url=result.get("image_url", ""),  # Go 端根据 content_type 计算
        )

        # 5. 最后 yield tool result text（喂回 LLM）
        yield (
            f"Image generated successfully.\n"
            f"- Filename: {filename}.png\n"
            f"- File ID: {result['file_id']}\n"
            f"The image has been displayed to the user on the canvas."
        )

        # 6. yield 图片生成的 token 用量，供上层累加到 TokenUsage
        if image_usage:
            yield ImageGenUsage(
                input_tokens=image_usage.input_tokens,
                output_tokens=image_usage.output_tokens,
            )

    except openai.BadRequestError as e:
        # GPT Image 1 内容审核拒绝（content_policy_violation）
        yield "Error: The image could not be generated because the prompt violates content policy. Please revise your description and try again."

    except openai.APITimeoutError:
        yield "Error: Image generation timed out (60s). Please try again."

    except httpx.HTTPStatusError as e:
        # Go 注册接口返回非 2xx（429 限流 / 配额不足等）
        if minio_written:
            asyncio.create_task(_cleanup_minio_file_with_retry(minio_path))
        error_msg = _parse_register_error(e)
        yield f"Error: {error_msg}"

    except asyncio.CancelledError:
        # Task 被取消（sse-starlette 主动取消），清理已写入的 MinIO 文件
        # 注意：即使 minio_written=False 也尝试清理，因为 asyncio.to_thread 可能内部已完成写入
        # 但 Python 层面的 await 被取消导致 minio_written 未被设为 True
        if minio_path:
            asyncio.create_task(_cleanup_minio_file_with_retry(minio_path))
        raise  # 重新抛出，让 asyncio 正常处理取消

    except Exception as e:
        # MinIO 写入失败或其他未知错误 → 仅在已成功写入时回滚
        if minio_written:
            asyncio.create_task(_cleanup_minio_file_with_retry(minio_path))
        logger.exception(f"create_image failed: {e}")
        yield "Error: Failed to generate image due to an internal error. Please try again."
```

```python
streaming_tools = [b for b in allowed_blocks if b.name == "create_image"]
normal_tools = [b for b in allowed_blocks if b.name != "create_image"]
```

**`_sanitize_filename` — 清洗 LLM 生成的文件名：**

```python
import re

def _sanitize_filename(filename: str) -> str:
    """移除路径分隔符和特殊字符，防止路径注入。保留空格和 Unicode 字符（中文等）以保持文件名可读性。"""
    # 只保留文件名部分，去掉所有目录路径
    filename = filename.rsplit("/", 1)[-1].rsplit("\\", 1)[-1]
    # 移除非法字符，只保留字母、数字、下划线、连字符、点、空格
    # 注意：\w 在 Python 中默认匹配 Unicode 字符（中文、日文等），这是有意行为
    # MinIO 和 DB 均原生支持 UTF-8 路径/文件名
    filename = re.sub(r'[^\w\-. ]', '_', filename)
    # 合并连续空格
    filename = re.sub(r' +', ' ', filename).strip()
    # 防止空文件名
```

**`_cleanup_minio_file_with_retry` — 清理孤儿文件（Go 注册失败时回滚，异步退避重试）：**

> **注意**：`minio_client` 的操作（`write_file`、`delete_file`）是同步阻塞调用，需要通过 `asyncio.to_thread()` 包装。清理函数为 async，内部包含退避重试逻辑（最多 2 次重试），调用方通过 `asyncio.create_task()` fire-and-forget 调度，不阻塞用户响应。
>
> **⚠️ 前置依赖**：当前 `minio_client` 只有 `read_file`、`write_file`、`read_file_as_text`、`write_text_file` 四个方法，**没有 `delete_file` 方法**。15.2 实现前必须先在 `services/minio_client.py` 中新增 `delete_file(path: str)` 方法，否则孤儿文件清理将直接失败。

```python
async def _cleanup_minio_file_with_retry(minio_path: str, max_retries: int = 2) -> None:
    """删除已写入 MinIO 但未成功注册的孤儿文件。异步退避重试，最多重试 2 次。
    调用方通过 asyncio.create_task() fire-and-forget，不阻塞用户响应。"""
    for attempt in range(max_retries + 1):
        try:
            await asyncio.to_thread(minio_client.delete_file, minio_path)
            logger.info(f"Cleaned up orphan file: {minio_path}")
            return
        except Exception as e:
            if attempt < max_retries:
                delay = 2 ** attempt  # 1s, 2s
                logger.warning(f"Failed to cleanup orphan file {minio_path} (attempt {attempt + 1}/{max_retries + 1}), retrying in {delay}s: {e}")
                await asyncio.sleep(delay)
            else:
                logger.warning(f"Failed to cleanup orphan file {minio_path} after {max_retries + 1} attempts, giving up: {e}")
```

**Redis 频率限制 Lua 脚本（Go 端实现）：**

```lua
-- KEYS[1] = "ai_file_gen:rate_limit:{userID}"
-- ARGV[1] = 86400 (TTL)
-- 返回值语义：正数 = 当前计数（1-10 表示成功），-1 = 已达上限被拒绝
local current = redis.call("GET", KEYS[1])

    return -1
end
local count = redis.call("INCR", KEYS[1])
if count == 1 then
    redis.call("EXPIRE", KEYS[1], ARGV[1])
end
return count
```

```go
func (s *FileService) checkRateLimit(ctx context.Context, userID int64) error {
    key := fmt.Sprintf("ai_file_gen:rate_limit:%d", userID)
    count, err := s.redis.Eval(ctx, rateLimitScript, []string{key}, 86400).Int()

    if count < 0 {
```

## 五、安全与限制

| 限制项 | 值 | 说明 |
|--------|-----|------|
| 24h 文件生成上限 | 10 次 | Redis Lua 脚本原子操作（INCR + 条件 EXPIRE 86400），**固定窗口**（从首次生成起算 24h），每个用户独立计数 |
| 用户总存储配额 | 200 MB | 复用现有 `maxStoragePerUser`，AI 生成的文件计入 |
| 图片生成 API 超时 | 60s | 图片生成较慢，超时比 web_search (15s) 放宽 |
| 文件生成 API 超时 | 15s | 文本文件只需写 MinIO + 注册，较快 |
| Tool Call Loop 总超时 | 120s | 复用现有限制 |
| 最大 Tool Call 轮数 | 10 | 复用现有限制 |
| 每轮最大 Tool Calls | 3 | 复用现有限制 |
| 每次用户轮次最多文件生成次数 | 1 | `create_image` 和 `create_file` 合计最多 1 次（跨轮累计，从用户发一条消息到 assistant 最终回复的完整轮次），tool description 提示 + llm.py `file_gen_used` 标志位跨轮强制执行 |
| AI 生成图片压缩 | 不压缩 | 保持最佳质量，配额 + 频率限制足够防滥用 |
| AI 生成文本文件大小 | 11MB 硬上限 | LLM 输出通常 <100KB，11MB 为防御性兜底；tool description 提示 AI 内容不超过 10MB |
| create_file 允许的扩展名 | .svg .md .csv .json .txt | 白名单控制，不允许生成可执行文件；HTML 不支持（用户需要 HTML 时 AI 以代码块形式输出） |
| 内部接口鉴权 | X-Internal-Token | 防止外部直接调用注册接口 |
| image_partial SSE 带宽 | ~200-400KB/张 × 2 张（JPEG q80 压缩后） | 原始 PNG 1-4MB 压缩为 JPEG q80；两跳 SSE 透传（ai-service → Go → 前端） |

### 部署注意事项

**ai-service 和 Go backend 必须同步部署（或 Go 先部署）。** 15.2 中 ai-service 将 `image_partial` / `resource_created` 的 SSE 格式从扁平改为嵌套（`{"type": "...", "data": {...}}`），Go 端需同步调整提取 `data` 字段的逻辑。如果 ai-service 先于 Go 部署，Go 仍按扁平格式处理嵌套 JSON，会导致前端收到双层嵌套的异常数据。

## 七、关键设计决策索引

| 决策项 | 选择 | 详见 |

| 流式生图 | stream + partial_images=2 + JPEG q80 压缩 | §2.3 |

| 职责分离 | ai-service 写 MinIO，Go 写 DB（事务） | §2.4, §2.6 |

| 生图费用追踪 | token 折算，累加到 assistant message 统计 | §2.10 |
| 频率限制 | Redis Lua 脚本（先 GET 判断再 INCR，超限返回 -1），固定窗口 24h/10次 | §2.13 |
| AI 图片不压缩 | 保持原始质量，配额+频率双重约束 | §2.13 |
| 文本文件大小 | 11MB 硬上限，tool description 提示 10MB | §2.13, §3.1.2 |
| 每次用户轮次文件生成次数 | `create_image` + `create_file` 合计限 1 次（跨轮累计，`file_gen_used` 标志位），被跳过的 block 仍返回 tool_result 提示信息 | §2.2, §3.1.2 |

| Go 端 image_partial | 提取嵌套 `data` 字段后透传，不 re-parse 内部字段 | §2.8, §2.15.2 |
| 错误处理 | 不重试，LLM 告知用户；Go 注册失败 → 清理 MinIO（minio_written 标志）；CancelledError → minio_path 非空即尝试清理 | §2.14 |
| 消息图片持久化 | 方案 B：image_url 由内部接口返回 → SSE 透传 → conversationService 在 complete/writeErrAssistant 时一并写入 message（避免流式期间 UPDATE 时序问题） | §2.16 |

| ResourceNode Handle | 复用现有 ResourceNode，加 target handle（`isConnectable={false}` + 隐藏），不新增 node_type | §2.12.1, §3.3.3 |
