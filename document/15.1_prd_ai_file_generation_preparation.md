# 15.1 AI 文件生成 — 前置改造（Preparation）

## 一、目标

对 ai-service 的 Tool Executor 返回类型、LLM 流式处理链路中的 tool 执行段进行结构性改造，同时在 Go backend 和前端做必要的适配，为 15.2 中 `create_image` / `create_file` 工具的接入打好基础。

**完成 15.1 后的期待效果：**
- 所有现有功能（web_search、url_reader、普通对话、SSE 流式回复、retry）**完全正常运行，行为无任何变化**
- ai-service 的 tool executor 已支持新的返回类型 `ToolResult` 和事件冒泡机制
- llm.py 三个 provider 函数的 tool 执行段已拆分为 streaming / non-streaming 两条路径
- Go backend 已能透传新的 SSE 事件类型（`image_partial`、`resource_created`），并在请求中携带 `ToolContext`
- 前端已注册新的 SSE 回调和 Redux reducers（但 15.1 阶段不会被触发）

---

## 二、背景：为什么需要改造

15.2 将新增两个 tool：
- `create_image`：流式生图，执行过程中需要向前端 **yield 中间事件**（partial image 预览）
- `create_file`：非流式，执行完成后需要返回 **附带事件**（resource_created）

**现有架构的限制：**

```python
# 现状：tool_executor.execute() 返回 str
async def execute(name: str, arguments: dict) -> str:
    ...

# 现状：llm.py 中 asyncio.gather 并行执行所有 tool
results = await asyncio.gather(*[
    tool_executor.execute(block.name, block.input)
    for block in tool_use_blocks
])
```

问题：
1. `execute()` 只返回 `str`，无法携带 side event（如 `ResourceCreatedEvent`）
2. `asyncio.gather()` 等所有 tool 完成才返回，无法在执行过程中 yield 中间事件（如 `ImagePartialEvent`）
3. `stream_chat()` 没有接收 `tool_context`（user_id、canvas_id、chat_node_id）的能力

**15.1 的改造让这些限制不再存在，15.2 只需「填入」新 tool 的具体逻辑。**

## 三、ai-service 改造

### 3.1 tool_executor.py — 返回类型重构 + 新增 dataclass

#### 3.1.1 新增 dataclass 和类型定义

```python
from __future__ import annotations
from dataclasses import dataclass

@dataclass
class ToolContext:
    """由 Go backend 传入的执行上下文，15.2 中 create_image/create_file 使用。"""
    user_id: int
    canvas_id: int
    chat_node_id: str
    message_id: str  # assistant message ID，用于前端关联 image_partial / resource_created 到具体消息

@dataclass
class ImagePartialEvent:
    """流式生图中间预览（15.2 create_image 使用）"""
    b64_image: str       # base64 编码的 JPEG q80（已从原始 PNG 压缩）
    partial_index: int   # 第几张 partial (0, 1, ...)
    chat_node_id: str    # 关联的 ChatNode ID
    message_id: str      # 关联的 assistant message ID

@dataclass
class ResourceCreatedEvent:
    """文件最终落库完成（15.2 create_image / create_file 使用）"""
    file_id: str        # int64 雪花 ID，Go 端返回 string，全链路用 string 传输
    node_id: str
    edge_id: str
    filename: str
    content_type: str
    chat_node_id: str
    message_id: str     # 关联的 assistant message ID
    position: dict  # {"x": float, "y": float}
    image_url: str      # 仅图片类型非空（如 "/api/file/xxx"），Go conversationService 在 complete 时写入 message

ToolSideEvent = ImagePartialEvent | ResourceCreatedEvent
ToolResult = tuple[str, list[ToolSideEvent]]
```

#### 3.1.2 `execute()` 签名变更

**Before：**
```python
async def execute(name: str, arguments: dict) -> str:
    if name == "web_search":
        return await web_search(arguments.get("query", ""))
    if name == "url_reader":
        return await url_reader(arguments.get("url", ""))
    return f"Error: unknown tool '{name}'"
```

**After：**
```python
async def execute(name: str, arguments: dict, context: ToolContext | None = None) -> ToolResult:
    """Dispatch a non-streaming tool call. Returns (result_text, side_events)."""
    if name == "web_search":
        return await web_search(arguments.get("query", "")), []
    if name == "url_reader":
        return await url_reader(arguments.get("url", "")), []
    return f"Error: unknown tool '{name}'", []
```

**关键点：**
- `web_search` 和 `url_reader` 返回 `(str, [])`，空事件列表，**行为完全不变**
- `context` 参数可选，15.1 阶段 llm.py 传 `None`
- 15.2 中新增的 `create_file` case 也在此函数中

#### 3.1.3 新增 `_compress_preview()` 辅助函数

```python
import base64
from io import BytesIO
from PIL import Image

def _compress_preview(b64_png: str, quality: int = 80) -> str:
    """PNG base64 → JPEG q80 base64，用于 partial image 预览压缩。
    原始 PNG 1-4MB → 压缩后 ~200-400KB。
    """
    img = Image.open(BytesIO(base64.b64decode(b64_png)))
    buf = BytesIO()
    img.convert("RGB").save(buf, format="JPEG", quality=quality)
    return base64.b64encode(buf.getvalue()).decode()
```

15.1 阶段此函数不会被调用，但需要确保 import 正常。

> **备注**：`from PIL import Image` 放在文件顶部。`Pillow==11.1.0` 已被现有文件转换功能（PyMuPDF、python-docx 等模块）使用并确认安装，不存在 import 失败风险。

#### 3.2.4 `stream_chat()` 签名扩展

**Before：**
```python
async def stream_chat(
    messages: list[dict], model_index: int
) -> AsyncGenerator[str | TokenUsage | ToolCallEvent, None]:
```

**After：**
```python
from services.tool_executor import ImagePartialEvent, ResourceCreatedEvent, ToolContext

async def stream_chat(
    messages: list[dict], model_index: int, tool_context: ToolContext | None = None
```

`tool_context` 传入三个 provider 函数，provider 函数内部把它传给 `tool_executor.execute()` 和 `tool_executor.create_image_stream()`。

15.1 阶段 `tool_context` 为 `None`（Go 不传或传 null）。

**关于 `stream_chat()` 函数体：** `stream_chat()` 内部通过 `async for item in gen: yield item` 将 provider 函数 yield 的所有值原样透传给调用方。因此 **不需要** 在 `stream_chat()` 内部新增 `isinstance` 判断 —— 只需扩展 yield 类型声明即可，provider 函数 yield 的 `ImagePartialEvent` / `ResourceCreatedEvent` 会自动透传到 `chat.py` 的 `event_generator` 中处理。

### 4.3 conversationService 构造 `StreamChatReq` 并传入 `ToolContext`

`internal/service/conversationService.go` 中 `SendMessage` 和 `RetryMessage` 均需构造完整 `StreamChatReq`：

```go
chatReq := infra.StreamChatReq{
    Messages: chatMessages,
    Model:    model,
    ToolContext: &infra.ToolContext{
        UserID:     userID,
        CanvasID:   canvasID,
        ChatNodeID: req.ConversationID, // conversationID == chatNodeID（一一对应，不同表）
```

`chatNodeID` 的值与 `conversationID` 完全一致（一一对应关系）。每个 Conversation 在创建时绑定一个 canvas chatNode，两者 ID 相同但分属不同表（conversations 表 vs canvas nodes 表）。因此直接使用 `req.ConversationID` 即可。

### 4.4 `AIStreamEvent` 结构体扩展 + Scanner 缓冲区

#### 4.4.1 问题：现有结构体无法承载新事件数据

现有 `SSEEvent` 和 `AIStreamEvent` 只有 `Type`、`Content`、`Message`、`PromptTokens`、`CompletionTokens` 等固定字段，无法携带 `image_partial` 的 `b64_image` 或 `resource_created` 的 `file_id` 等字段。

**解决方案：给 `AIStreamEvent` 新增 `RawData` 字段，用于原样透传未知结构的事件数据。**

```go
type AIStreamEvent struct {
    Type             string          // "token" | "complete" | "error" | "tool_call" | "image_partial" | "resource_created"
    Content          string          // token content or full reply
    PromptTokens     int
    CompletionTokens int
    RawData          json.RawMessage // 新增：原始 JSON data，用于 image_partial / resource_created 透传
}
```

#### 4.4.2 问题：Scanner 默认缓冲区不足

现有 `StreamChat` 使用 `bufio.NewScanner` 解析 SSE 流，**默认缓冲区上限为 64KB**（`bufio.MaxScanTokenSize`）。15.2 中 `image_partial` 事件携带 200-400KB 的 base64 字符串，单行 SSE data 将**远超 64KB 限制**，导致 scanner 报错断流。

**解决方案：15.1 阶段就扩大 scanner 缓冲区至 2MB。**

```go
scanner := bufio.NewScanner(resp.Body)
scanner.Buffer(make([]byte, 2*1024*1024), 2*1024*1024) // 2MB buffer，支持大 base64 payload
```

虽然 15.1 阶段不会触发大 payload，但既然是"搭管道"，管道口径应一步到位。

#### 4.4.3 SSE 事件透传

`infra/aiClient.go` 的 `StreamChat` SSE 解析新增两个 case：

```go
case "image_partial", "resource_created":
    // 不做 JSON re-parse，直接将原始 data 赋值给 RawData 透传
    out = AIStreamEvent{Type: event.Type, RawData: json.RawMessage(line[6:])} // line 是 scanner.Text()，line[6:] 去掉 "data: " 前缀后的原始 JSON
```

`conversationService.go` 的 `SendMessage` 和 `RetryMessage` 中 `for evt := range aiCh` 均需新增两个 case，直接将 `RawData` 写入前端 SSE response writer：

```go
case "image_partial", "resource_created":
    eventCh <- dto.SSEEvent{
        Type: event.Type,
        Data: json.RawMessage(event.RawData), // 原样透传，不反序列化
    }
```

**注意：`SendMessage` 和 `RetryMessage` 两个函数中的事件处理循环都需要新增这两个 case。**

**Handler 层**也需要适配：当 `SSEEvent.Data` 已经是 `json.RawMessage` 时，`json.Marshal` 不会重复编码，直接嵌入原始 JSON。

15.1 阶段 ai-service 不会产生这两种事件，所以这两个 case 不会被触发。
